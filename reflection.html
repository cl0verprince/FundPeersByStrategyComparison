Reflection — decision log

Why these techniques

Random forest for the underperformance prediction (over logistic regression, gradient boosting, or neural networks)

The problem is a small-to-mid-size tabular panel (~24k fund-quarter rows, a dozen features of mixed scales) with a noisy financial label - exactly where tree ensembles are the professional default. A forest captures nonlinear interactions that plainly exist here (a deep drawdown means something different for a concentrated sector fund than for a diversified large blend) without feature scaling or distributional assumptions, and bagging across decorrelated trees tames the overfitting a single tree suffers on noisy labels - measured directly in this project: single tree AUC 0.638 vs forest 0.710 on the same split. Feature importances come natively, which matters for a financial audience that needs to see what drives a prediction. A neural network has no advantage at this data size and costs the explainability; logistic regression can't represent the interactions; gradient boosting is a reasonable alternative but is more sensitive to tuning and to label noise, without a demonstrated payoff here.

A single decision tree fitted first, then the forest

Deliberate progression, not redundancy: the interpretable single tree establishes that human-readable rules with real signal exist in the features at all (its splits can be printed and sanity-checked by a person), and the forest then shows how much additional accuracy comes purely from variance reduction. Reporting both separates 'is there signal?' from 'how much does ensembling buy?'.

A local LLM (phi-4, quantized GGUF, served by LM Studio) instead of a hosted API

Three professional reasons. Cost/independence: the plan explicitly excludes paid vendors, and a local model has zero marginal cost, no rate limits, and no API keys to manage. Reproducibility: pinned quantized weights plus temperature 0 make narration re-runnable and auditable, consistent with the project's deterministic-conductor requirement - a hosted API's model can be silently upgraded under you, changing outputs between runs. Fit-for-purpose: the LLM's only job here is turning already-computed numbers into plain English, a task a 14B local model does well; nothing in the pipeline needs frontier-model reasoning.

RAG-style grounding for all LLM narration (retrieve computed facts, inject them into the prompt, forbid outside knowledge)

LLMs fabricate plausible-sounding numbers when asked to recall facts - unacceptable in financial reporting. The narration prompt therefore contains every fact the model is allowed to use (the fund's metrics, peers, cluster stats, prediction - all retrieved from the pipeline's own verified tables) and instructs it to use nothing else. This makes every sentence auditable against a table, and it draws the division of labor cleanly: classical ML produces every number and decision; the LLM only verbalizes. Verified in practice - a full narration was spot-checked claim by claim against the retrieved context, with zero invented numbers or peers.

Sentence embeddings of a holdings-text description (all-MiniLM-L6-v2) as the fund similarity representation

Chosen by measurement, not preference. The obvious approach - vectors of raw issuer overlap - was built first and scored at chance level against independent Yahoo categories (adjusted Rand index 0.01-0.07, insensitive to every tuning knob), because popular mega-caps are held across all styles, so raw overlap mostly measures 'which companies are popular'. Embedding a text description of each fund's top holdings raised ARI to ~0.25, a ~10x improvement, while running fully local and fast (~60s for 12 quarters). The small sentence-transformer is sufficient because the input is a short, structured holdings list, not open prose.

KMeans on L2-normalized embeddings for peer-group clustering, with cosine similarity for nearest-peer lookup

On L2-normalized vectors, euclidean KMeans is equivalent to clustering by cosine direction, so the cluster geometry matches the similarity measure used everywhere else in the pipeline. KMeans is deterministic under a fixed seed (a project-wide requirement), fast enough to re-fit per quarter, and its one real knob - the cluster count - was chosen from a measured sweep (chance-corrected category agreement peaking, cluster-size floor for statistical power) rather than picked by feel.

Time-based train/test split (train 2022-2023, test 2024), never a random split

Financial panel rows are autocorrelated within a fund and within a quarter; a random row split would let the model train on a fund's 2024 behavior and be tested on the same fund's 2023 behavior - look-ahead leakage that inflates accuracy and evaporates in deployment. Training strictly on the past and testing strictly on the future mimics how the model would actually be used, and the split is asserted in code (max train quarter < min test quarter) rather than trusted.

AUC against explicit baselines as the headline metric

AUC measures ranking quality without committing to a threshold - the natural fit for the product's real use (rank funds by underperformance risk within a peer group; different users would act at different cutoffs), and it is insensitive to the modest class imbalance that accuracy would reward gaming. It is reported against explicit baselines (coin-flip 0.5; a persistence rule for the unified model) because an AUC with no baseline invites over-reading - the number only means something relative to what a naive strategy achieves.

Cluster validation against an independent external label (Yahoo categories, via purity and adjusted Rand index)

Internal cohesion metrics (inertia, silhouette) can be optimized by any clustering, meaningful or not - they never see ground truth. Scoring against fund categories assigned by an independent third party is an external check the clustering cannot game, and using the adjusted Rand index specifically corrects for chance agreement, so a larger cluster count cannot inflate the score mechanically the way raw purity can. Yahoo's categories are treated as a noisy heuristic validator, not as truth - which is exactly the right epistemic weight for an unofficial data source.

kNN-peer median (each fund's own top-10 most-similar funds) as the underperformance benchmark in the unified rebuild, replacing the cluster median

A label must mean the same thing at every population size. 'Below your cluster's median' silently degraded as the universe grew - the same 15 clusters that meant ~12 close peers per fund at 363 funds meant a median of ~62 heterogeneous funds at 1,489, measurably diluting both the label and the model's top feature. Defining the benchmark as each fund's own top-10 cosine-similarity peers makes the peer group bespoke and constant-size by construction, so the prediction task no longer changes definition as more funds are ingested - the root-cause fix, where re-tuning the cluster count each time would be a recurring patch.

Decision log

2026-07-08

Use SEC bulk Form N-PORT Data Sets (quarterly ZIPs of tab-delimited files) as the primary data source.

Verified against SEC primary sources: pre-parsed, joinable flat files covering fund identity, monthly returns, and holdings. Avoids scraping individual XML filings entirely.

2026-07-08

Identify the US equity fund universe using Yahoo Finance categories as a pre-filter, cross-checked against holdings-based inference (value-weighted ASSET_CAT='EC' + INVESTMENT_COUNTRY='US').

N-PORT has no fund-level equity/strategy label, so it must be inferred from holdings. Yahoo categories (joined via SEC's official mutual-fund ticker map) give a fast, independent cross-check, but are treated as a heuristic since yfinance is an unofficial scraper.

2026-07-08

Build peer groups ourselves via cosine similarity + clustering over holdings-based strategy vectors, then validate the clusters against Yahoo fund categories (purity / adjusted Rand index).

Yahoo categories could have been used directly as peer groups, but that would skip building real holdings-based similarity/clustering. Using them as a validator instead keeps the clustering work while still getting an external sanity check.

2026-07-08

Define 'underperform' as a fund's next-quarter total return falling below its cluster's median return, evaluated across multiple pooled quarterly prediction periods (not a single train/test split).

N-PORT holdings snapshots are effectively quarterly (only the 3rd month of each quarter is historically public), but each filing reports all 3 monthly returns — this maps cleanly onto a quarter-over-quarter prediction setup with several periods pooled for a realistic time-based split.

2026-07-08

Use phi-4 (Q4_K_S) served locally by LM Studio for plain-English narration of results only (RAG-style, grounded in computed data) — not for training or fine-tuning.

The quantized GGUF model cannot be trained, and the core data (fund holdings/returns) is numeric — a poor fit for LLM prediction. The LLM's strength here is explaining results already produced by classical ML, not producing the predictions itself.

2026-07-08

Run the project as a step-wise-methodology build: a contract-first plan.md, one gated milestone at a time, each with its own design.md and a human approval gate before the next step starts.

Keeps each stage reviewable and reversible, and avoids docs drifting from code since decisions/workflow docs update in the same change as each step.

2026-07-09

Set the fund universe target to 1000 resolved funds, using an iterative-fill sampling strategy (shuffle the full complete-panel candidate pool, resolve ticker+Yahoo data one at a time until 1000 succeed) rather than a fixed pre-filter sample.

A 20-fund pilot run showed ~45% of candidate series have no ticker mapping at all before Yahoo is even attempted; a fixed sample of exactly max_funds candidates would under-shoot the target after attrition. The complete 12-quarter panel has 9,445 candidate series, giving ample headroom.

2026-07-09

Corrected the US-equity flag to require BOTH yahoo_stock_position >= 0.80 (asset class) AND a holdings-based US-domiciled-equity share >= 0.70 (geography), instead of the asset-class signal alone.

Empirically confirmed yfinance's stockPosition only measures stock-vs-bond allocation, not geography - international and emerging-market equity funds also show stockPosition near 1.0. Combining it with a value-weighted holdings check (ASSET_CAT='EC' + INVESTMENT_COUNTRY='US') fixed 87/90 misflagged international funds in testing; the 3 remaining exceptions trace to a documented, accepted N-PORT data limitation (fund-of-funds wrappers reporting their issuer's country, not their economic exposure).

2026-07-09

Rejected category-only feature vectors (asset_cat/issuer_type/country) for fund clustering, in favor of building strategy vectors from actual portfolio overlap - since the step2 universe is already equity-filtered, category fields alone are nearly identical across all 363 funds and carry almost no discriminative signal.

Caught before implementation: clustering on near-identical category vectors would produce a degenerate result (one dominant cluster), defeating the purpose of peer grouping.

2026-07-09

Built and empirically tested issuer-overlap vectors (value-weighted, then breadth-weighted top-K held issuers) for clustering, then rejected them in favor of local text-embeddings of a holdings description (sentence-transformers, all-MiniLM-L6-v2).

Issuer-overlap vectors scored near chance-level agreement with Yahoo fund categories (adjusted Rand Index 0.01-0.07, confirmed insensitive to K and to cluster count 5-60 - ruled out as a tuning problem). Embeddings of a text description of each fund's top holdings raised ARI to ~0.25, roughly a 10x improvement, while running fully locally and fast (~60s for the full 12-quarter pipeline). Root cause of the overlap approach's failure: popular mega-cap stocks are held broadly across growth/value/blend categories, so raw stock overlap mostly captures 'which companies are popular' rather than value/growth style, which text embeddings apparently capture better.

2026-07-10

Added a GitHub Actions workflow (.github/workflows/tests.yml) running the fast pytest suite on every push/PR to main. Confirmed no CI/CD existed previously (no .github/workflows directory, and 3 commits sat unpushed on origin before this).

The 10 unit/smoke tests are fast (~5s) and need no network access or secrets (verified by running them with the data/ directory temporarily removed, simulating a clean checkout), making them well-suited to CI. The full pipeline (step1 ingestion: ~15+ min, GBs of downloads, needs SEC_USER_AGENT and live external APIs) deliberately stays out of CI and is only run manually per gated step, per the step-wise-methodology approach used throughout this project.

2026-07-10

Fixed a real bug in compute_quarterly_returns: np.prod() on a pandas Series silently skips NaN via Series.prod(skipna=True), so a fund with one unparseable monthly return (2 funds affected, from step1's pd.to_numeric coercion) got a silently-wrong 2-of-3-month product for that quarter instead of a correctly-NaN result.

Caught by cross-checking .describe() row counts between fund_metrics_overall (correctly NaN via .values, a numpy array) and fund_metrics_quarterly (incorrectly non-NaN) for the same 2 funds - the discrepancy revealed the inconsistency. Fixed by using .values in the quarterly calc too, with a regression test added.

2026-07-10

Migrated the persistence layer from standalone .parquet files to a single embedded SQL database (DuckDB, data/processed/fundspeers.duckdb), on request. PostgreSQL was considered and rejected in favor of DuckDB.

PostgreSQL needs a running server process and connection credentials - real new infrastructure this project doesn't otherwise have. DuckDB is serverless (a single file, no credentials) and gives the same real-SQL experience (joins, aggregates, window functions, any DuckDB client). Migrated existing parquet output directly into the new file (no recompute needed) and confirmed every step's code needed zero changes, validating the fundspeers.io abstraction built in step0.

2026-07-10

Built step4's features as trailing (point-in-time-safe) metrics computed fresh from monthly_returns, rather than reusing step3's fund_metrics_overall - and used step2's category-tier bucketing (not raw cluster_id) as the only cross-quarter-comparable categorical feature.

fund_metrics_overall is computed from a fund's ENTIRE 36-month history, so using it as a feature 'at quarter Q' would leak information from quarters after Q. Separately, step2's cluster_id is an arbitrary per-quarter KMeans label with no meaning across quarters, so pooling it as a feature across the 11 quarterly transitions would be meaningless; yahoo_category (bucketed into Large/Mid/Small/Sector tiers) is stable across quarters and used instead. Both risks were designed around before writing code, then verified with dedicated unit tests (point-in-time-safety, split leakage) rather than discovered afterward.

2026-07-10

Random forest (AUC 0.710) beats both the coin-flip baseline (0.500) and a single decision tree (0.638) on next-quarter peer-underperformance, using a time-based split (train on 2022-2023, test on 2024) - confirmed on real data, not just the Required Output's minimum bar.

Feature importances are dominated by the fund's own recent performance (return_vs_cluster_median_q, trailing drawdown/Sharpe/return) rather than fund size or category tier, which is a plausible pattern (not a sign of a spurious or leaking feature).

2026-07-10

step5_narrate generates narrations on demand (via narrate_fund()) rather than precomputing a persisted table for all 4,356 fund-quarters; the conductor's run() only narrates one illustrative example (the highest predicted-underperformance-probability fund in the test split).

There's no requirement to narrate the whole panel, and doing so would be slow/costly for a local model with no clear use for the bulk output. Verified end-to-end against the live LM Studio server (confirmed running phi-4 at http://127.0.0.1:1234/v1): every claim in a real narration (ticker, category, all four whole-history metrics, quarterly return, cluster id, purity/ARI, two named peers, predicted probability, actual outcome) was spot-checked against the raw retrieved context and matched exactly - no invented numbers or peers. Both documented edge cases (no prediction available, no resolved cluster) were tested directly and handled gracefully.

2026-07-11

Added hard (deterministic, rule-based) cluster definitions - dominant category/tier and average risk metrics per (quarter, cluster_id) - rather than an LLM-generated description, keeping clusters' numeric ids as their only label until now.

Clusters previously had no human-readable identity at all, just an arbitrary per-quarter number. A rule-based definition (majority category, average Sharpe/volatility) keeps the same LLM boundary already established in step5 - the model narrates already-computed facts, it never labels or decides anything itself. Lives in step3 (needs fund_metrics_overall, which doesn't exist until step3 runs) even though it describes step2's clusters. Needing category_tier a third time here (after step2 and step4) triggered extracting it into a shared fundspeers/category.py module instead of copy-pasting again - verified as a pure refactor by re-running step2 and confirming byte-identical purity/ARI results.

2026-07-11

Added step6_out_of_sample: sampled 1000 new funds fully disjoint from the original training set, ran the same pipeline (ingest/similarity/metrics) against a parallel '_oos' table namespace, and evaluated the step4 model two ways - frozen (never refit) and retrained on the combined original+OOS set.

Both were requested and both answer different questions. Frozen-model AUC on 1000 new never-trained-on funds was 0.693 vs. the original 0.710 test AUC - only a modest drop, real evidence the model generalizes to new funds rather than memorizing quirks of the original 363. Retrained-on-combined AUC was 0.725, a further modest improvement consistent with more diverse data helping a little. ingest.py/similarity.py/metrics.py's run() functions gained backward-compatible exclude_series/table_suffix parameters (defaults preserve original behavior exactly, verified via a small-scope disjointness test before the full 1000-fund run) rather than duplicating the pipeline. step4 now persists the fitted model (joblib) bundled with its exact training feature-column list, since nothing had saved the model object itself before - only its predictions.

2026-07-11

Added cluster_definitions.short_title, a compact allocation-only bucket name (e.g. "Leaning Large Blend", "Concentrated Energy Limited Partnership") - built from dominant_category_share and dominant_category only, deliberately excluding any performance stat (avg_sharpe, avg_volatility).

User caught two real problems with an earlier draft that included a performance word: (1) clusters are formed from holdings/allocation similarity, not performance, so baking a performance stat into the identifying name implies the cluster was formed because of its performance, which isn't true; (2) a performance band worded as "Underperforming" collides with this pipeline's existing, different meaning of underperform (relative to a fund's own cluster median, from step4) - a cluster can't underperform itself. The longer title field still reports avg_volatility/avg_sharpe as descriptive stats, just not folded into the cluster's identity.

2026-07-11

Extracted concentration_word/compute_dominant_category_info into shared fundspeers/category.py (used by both step2's cluster-map legend and step3's cluster_definitions table), and replaced the cluster map's numeric cluster_id colorbar with a proper legend showing each cluster's short_title.

short_title only needs dominant_category/dominant_category_share, not the metrics-dependent fields step3 also computes, so it can be (and now is) computed directly in step2 before step3 ever runs - avoiding a third near-duplicate of the same logic. Comparing the original and OOS cluster maps with real names visible (not just colors) made a genuine structural consistency check possible: the separated left-island cluster in both plots is dominated by Target-Date allocation funds, even though the arbitrary cluster_id differs between the two independent KMeans fits.

2026-07-11

Confirmed the original-vs-OOS cluster label mismatch (different colors/ids for the same visual island) is expected, not a bug: both similarity.py's KMeans and PCA are freshly re-fit on each run's own data, never reusing the other run's fitted centroids/transform - and the two runs use fully disjoint funds anyway, so ARI/AMI between them isn't even computable.

User asked directly whether OOS was re-clustered independently or scored against the existing fitted model; verified from the code (KMeans(...).fit_predict(...) inside the per-quarter loop, called fresh every run()) rather than assumed. The right structural check is each run's own purity/ARI against Yahoo categories (already computed), not cross-run label agreement.

2026-07-11

Promoted the retrained-on-combined model (754 funds, test AUC=0.725) to be the new official models/random_forest_model.joblib, replacing the original 363-fund model - backed up to random_forest_model_original_363funds.joblib rather than silently overwritten, and regenerated fund_predictions/model_feature_importances so the persisted tables stay consistent with whichever model is actually saved as official.

User explicitly opted in after being told the retrained model was never persisted before (only its AUC was reported) - promoting is a deliberate, explicit action, not an automatic side effect of running step6's evaluation.
