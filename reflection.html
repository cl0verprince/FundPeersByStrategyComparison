Reflection — decision log

Why these techniques

Random forest for the underperformance prediction (over logistic regression, gradient boosting, or neural networks)

The problem is a small-to-mid-size tabular panel (~24k fund-quarter rows, a dozen features of mixed scales) with a noisy financial label - exactly where tree ensembles are the professional default. A forest captures nonlinear interactions that plainly exist here (a deep drawdown means something different for a concentrated sector fund than for a diversified large blend) without feature scaling or distributional assumptions, and bagging across decorrelated trees tames the overfitting a single tree suffers on noisy labels - measured directly in this project: single tree AUC 0.638 vs forest 0.710 on the same split. Feature importances come natively, which matters for a financial audience that needs to see what drives a prediction. A neural network has no advantage at this data size and costs the explainability; logistic regression can't represent the interactions; gradient boosting is a reasonable alternative but is more sensitive to tuning and to label noise, without a demonstrated payoff here.

A single decision tree fitted first, then the forest

Deliberate progression, not redundancy: the interpretable single tree establishes that human-readable rules with real signal exist in the features at all (its splits can be printed and sanity-checked by a person), and the forest then shows how much additional accuracy comes purely from variance reduction. Reporting both separates 'is there signal?' from 'how much does ensembling buy?'.

A local LLM (phi-4, quantized GGUF, served by LM Studio) instead of a hosted API

Three professional reasons. Cost/independence: the plan explicitly excludes paid vendors, and a local model has zero marginal cost, no rate limits, and no API keys to manage. Reproducibility: pinned quantized weights plus temperature 0 make narration re-runnable and auditable, consistent with the project's deterministic-conductor requirement - a hosted API's model can be silently upgraded under you, changing outputs between runs. Fit-for-purpose: the LLM's only job here is turning already-computed numbers into plain English, a task a 14B local model does well; nothing in the pipeline needs frontier-model reasoning.

RAG-style grounding for all LLM narration (retrieve computed facts, inject them into the prompt, forbid outside knowledge)

LLMs fabricate plausible-sounding numbers when asked to recall facts - unacceptable in financial reporting. The narration prompt therefore contains every fact the model is allowed to use (the fund's metrics, peers, cluster stats, prediction - all retrieved from the pipeline's own verified tables) and instructs it to use nothing else. This makes every sentence auditable against a table, and it draws the division of labor cleanly: classical ML produces every number and decision; the LLM only verbalizes. Verified in practice - a full narration was spot-checked claim by claim against the retrieved context, with zero invented numbers or peers.

Sentence embeddings of a holdings-text description (all-MiniLM-L6-v2) as the fund similarity representation

Chosen by measurement, not preference. The obvious approach - vectors of raw issuer overlap - was built first and scored at chance level against independent Yahoo categories (adjusted Rand index 0.01-0.07, insensitive to every tuning knob), because popular mega-caps are held across all styles, so raw overlap mostly measures 'which companies are popular'. Embedding a text description of each fund's top holdings raised ARI to ~0.25, a ~10x improvement, while running fully local and fast (~60s for 12 quarters). The small sentence-transformer is sufficient because the input is a short, structured holdings list, not open prose.

KMeans on L2-normalized embeddings for peer-group clustering, with cosine similarity for nearest-peer lookup

On L2-normalized vectors, euclidean KMeans is equivalent to clustering by cosine direction, so the cluster geometry matches the similarity measure used everywhere else in the pipeline. KMeans is deterministic under a fixed seed (a project-wide requirement), fast enough to re-fit per quarter, and its one real knob - the cluster count - was chosen from a measured sweep (chance-corrected category agreement peaking, cluster-size floor for statistical power) rather than picked by feel.

Time-based train/test split (train 2022-2023, test 2024), never a random split

Financial panel rows are autocorrelated within a fund and within a quarter; a random row split would let the model train on a fund's 2024 behavior and be tested on the same fund's 2023 behavior - look-ahead leakage that inflates accuracy and evaporates in deployment. Training strictly on the past and testing strictly on the future mimics how the model would actually be used, and the split is asserted in code (max train quarter < min test quarter) rather than trusted.

AUC against explicit baselines as the headline metric

AUC measures ranking quality without committing to a threshold - the natural fit for the product's real use (rank funds by underperformance risk within a peer group; different users would act at different cutoffs), and it is insensitive to the modest class imbalance that accuracy would reward gaming. It is reported against explicit baselines (coin-flip 0.5; a persistence rule for the unified model) because an AUC with no baseline invites over-reading - the number only means something relative to what a naive strategy achieves.

Monte Carlo as the uncertainty layer around evaluation - deliberately NOT as a simulator of fund returns

Considered for the predictive model itself and rejected there: Monte Carlo simulates outcomes from a stochastic model you already assume, so simulating fund returns would mean assuming a return-generating process (plus cross-fund correlations, since the label is peer-relative) - replacing patterns measured from data with patterns assumed by the analyst. Where it genuinely earns its keep here is quantifying uncertainty in the evaluation: (1) a fund-clustered bootstrap turns the headline test AUC into an interval - essential honesty when the test window is only 3 quarters and per-quarter AUC was measured swinging 0.59-0.82; (2) a paired bootstrap on the same resamples tests whether the model's edge over the naive persistence rule is statistically real, the first question a skeptical analyst asks; (3) perturbing each fund's peer set and measuring label flip-rates estimates the benchmark's intrinsic noise floor, guarding against tuning the model into pure label noise. All three are seeded resampling procedures - deterministic, cheap (no refitting), and reported whichever way they come out.

Cluster validation against an independent external label (Yahoo categories, via purity and adjusted Rand index)

Internal cohesion metrics (inertia, silhouette) can be optimized by any clustering, meaningful or not - they never see ground truth. Scoring against fund categories assigned by an independent third party is an external check the clustering cannot game, and using the adjusted Rand index specifically corrects for chance agreement, so a larger cluster count cannot inflate the score mechanically the way raw purity can. Yahoo's categories are treated as a noisy heuristic validator, not as truth - which is exactly the right epistemic weight for an unofficial data source.

kNN-peer median (each fund's own top-10 most-similar funds) as the underperformance benchmark in the unified rebuild, replacing the cluster median

A label must mean the same thing at every population size. 'Below your cluster's median' silently degraded as the universe grew - the same 15 clusters that meant ~12 close peers per fund at 363 funds meant a median of ~62 heterogeneous funds at 1,489, measurably diluting both the label and the model's top feature. Defining the benchmark as each fund's own top-10 cosine-similarity peers makes the peer group bespoke and constant-size by construction, so the prediction task no longer changes definition as more funds are ingested - the root-cause fix, where re-tuning the cluster count each time would be a recurring patch.

2026-07-08

Use SEC bulk Form N-PORT Data Sets (quarterly ZIPs of tab-delimited files) as the primary data source.

Verified against SEC primary sources: pre-parsed, joinable flat files covering fund identity, monthly returns, and holdings. Avoids scraping individual XML filings entirely.

2026-07-08

Identify the US equity fund universe using Yahoo Finance categories as a pre-filter, cross-checked against holdings-based inference (value-weighted ASSET_CAT='EC' + INVESTMENT_COUNTRY='US').

N-PORT has no fund-level equity/strategy label, so it must be inferred from holdings. Yahoo categories (joined via SEC's official mutual-fund ticker map) give a fast, independent cross-check, but are treated as a heuristic since yfinance is an unofficial scraper.

2026-07-08

Build peer groups ourselves via cosine similarity + clustering over holdings-based strategy vectors, then validate the clusters against Yahoo fund categories (purity / adjusted Rand index).

Yahoo categories could have been used directly as peer groups, but that would skip building real holdings-based similarity/clustering. Using them as a validator instead keeps the clustering work while still getting an external sanity check.

2026-07-08

Define 'underperform' as a fund's next-quarter total return falling below its cluster's median return, evaluated across multiple pooled quarterly prediction periods (not a single train/test split).

N-PORT holdings snapshots are effectively quarterly (only the 3rd month of each quarter is historically public), but each filing reports all 3 monthly returns — this maps cleanly onto a quarter-over-quarter prediction setup with several periods pooled for a realistic time-based split.

2026-07-08

Use phi-4 (Q4_K_S) served locally by LM Studio for plain-English narration of results only (RAG-style, grounded in computed data) — not for training or fine-tuning.

The quantized GGUF model cannot be trained, and the core data (fund holdings/returns) is numeric — a poor fit for LLM prediction. The LLM's strength here is explaining results already produced by classical ML, not producing the predictions itself.

2026-07-08

Run the project as a step-wise-methodology build: a contract-first plan.md, one gated milestone at a time, each with its own design.md and a human approval gate before the next step starts.

Keeps each stage reviewable and reversible, and avoids docs drifting from code since decisions/workflow docs update in the same change as each step.

Set the fund universe target to 1000 resolved funds, using an iterative-fill sampling strategy (shuffle the full complete-panel candidate pool, resolve ticker+Yahoo data one at a time until 1000 succeed) rather than a fixed pre-filter sample.

A 20-fund pilot run showed ~45% of candidate series have no ticker mapping at all before Yahoo is even attempted; a fixed sample of exactly max_funds candidates would under-shoot the target after attrition. The complete 12-quarter panel has 9,445 candidate series, giving ample headroom.

Empirically confirmed yfinance's stockPosition only measures stock-vs-bond allocation, not geography - international and emerging-market equity funds also show stockPosition near 1.0. 2026-07-09

Built and empirically tested issuer-overlap vectors (value-weighted, then breadth-weighted top-K held issuers) for clustering, then rejected them in favor of local text-embeddings of a holdings description (sentence-transformers, all-MiniLM-L6-v2).

Issuer-overlap vectors scored near chance-level agreement with Yahoo fund categories (adjusted Rand Index 0.01-0.07, confirmed insensitive to K and to cluster count 5-60 - ruled out as a tuning problem). Embeddings of a text description of each fund's top holdings raised ARI to ~0.25, roughly a 10x improvement, while running fully locally and fast (~60s for the full 12-quarter pipeline). Root cause of the overlap approach's failure: popular mega-cap stocks are held broadly across growth/value/blend categories, so raw stock overlap mostly captures 'which companies are popular' rather than value/growth style, which text embeddings apparently capture better.
